# 日程与任务操作规范

当用户涉及日程、任务、时间安排相关请求时，按此规范执行。

## 1. 操作权限映射

| 用户表达 | 工具调用 |
|----------|----------|
| "标记完成""做完了""搞定了""完成了" | update_task（status=completed） |
| "取消任务" | update_task（status=cancelled） |
| "修改日程""改期""调整时间""改标题/地点" | update_event（传入 event_id 及要修改的字段：title、start_time、end_time、location 等） |

## 2. 过期任务处理

当查询任务时，若工具返回结果中 metadata 含 `has_overdue: true`，必须主动询问用户过期任务的完成情况。

**流程：**

1. **主动询问**：在展示任务列表后，说明有任务过期并列出标题和截止日期，询问用户是否已完成。
2. **根据用户回复处理**：
   - 已完成 → update_task（status=completed）
   - 需要继续 → 询问新截止日期，update_task 更新 due_date（格式：YYYY-MM-DD）
   - 取消 → update_task（status=cancelled）
   - 用户提供新日期 → 解析并更新 due_date
   - 未明确回复 → 可再次询问或等待指示

## 3. 行程状态更新的 Proactive 行为

当用户提到可能影响时间安排的信息（如「坐车回家」「晚点」「到家了」等）：

1. 查询今日日程，识别可能相关的已有事件
2. 若有关联，主动提议更新或直接执行更新后告知用户
3. 优先通过 get_events 查看今日安排，再决定是否更新

## 4. 任务完成时处理相关日程

当用户标记任务为已完成，且该任务有关联日程事件时：

1. **查找相关日程**：通过 get_events 查询与任务标题相关的日程事件
2. **区分处理**：
   - 已过期 → update_event（status=completed）
   - 未来事件 → delete_schedule_data 删除
3. **主动处理**：标记任务完成后主动检查并处理相关日程，向用户说明结果

## 5. 展示格式

向用户展示日程事件或待办任务时，必须使用 Markdown 表格格式。

**日程事件：**

| 序号 | 标题 | 时间段 | 优先级 | 标签 |
|------|------|--------|--------|------|

**待办任务：**

| 序号 | 标题 | 状态 | 优先级 | 预计时长 | 截止日期 |
|------|------|------|--------|----------|----------|

**格式规则：**

- 日期：MM-DD，跨年用 YYYY-MM-DD
- 状态：待办、进行中、已完成、已取消
- 优先级：低、中、高、紧急
